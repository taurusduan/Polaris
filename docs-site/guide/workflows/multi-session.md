# 实战：多会话协作

本文演示如何使用 Polaris 的多会话功能，同时处理多个开发任务。

## 为什么需要多会话

在日常开发中，经常需要同时处理多个任务：

- 修复一个 Bug 的同时，审查另一个 PR 的代码
- 让 AI 分析代码的同时，继续在另一个会话中编写文档
- 对比不同 AI 模型对同一个问题的回答

Polaris 支持最多 **16 个会话**同时展示。

## 基本操作

### 创建多个会话

1. 点击 AI 面板顶部的 **新建会话** 按钮
2. 选择主工作区和关联工作区
3. 重复创建多个会话

### 切换到多会话视图

在聊天底部的状态栏区域调整会话布局：

| 配置 | 说明 |
|------|------|
| 行数 | 1 行 或 2 行 |
| 格子宽度 | 自定义每个会话的显示宽度（250~450px） |
| 全屏 | 双击任意会话窗口可全屏查看 |

### 会话管理

| 操作 | 说明 |
|------|------|
| 新建会话 | 创建独立的新对话 |
| 快速切换 | 鼠标悬停面板顶部的快速切换按钮，查看所有会话列表 |
| 会话历史 | 打开历史面板，按时间分组查看所有历史会话 |
| 恢复会话 | 从历史中恢复之前的对话继续 |
| 导出会话 | 将对话导出为 Markdown 或 JSON 文件 |

## 实战场景

### 场景一：Bug 修复 + 代码审查并行

**会话 A**（左侧）：修复 Bug

```
请帮我修复 src/api/user.ts 中的登录验证 Bug
```

**会话 B**（右侧）：审查代码

```
请审查 src/components/Dashboard.tsx 的最新变更
```

两个会话独立运行，互不干扰。各自有独立的工具调用、权限审批和上下文。

### 场景二：对比不同模型的回答

**会话 A**：使用 Sonnet 模型（快速响应）

**会话 B**：使用 Opus 模型（深度思考）

对同一个问题分别在两个会话中提问，对比结果质量。

> 在每个会话底部的状态栏中可以独立设置 **Model**（Sonnet / Opus / Haiku）。

### 场景三：跨工作区协作

1. 创建会话 A，主工作区设为 `frontend`，关联工作区添加 `backend`
2. 创建会话 B，主工作区设为 `docs`，独立处理文档任务
3. AI 在会话 A 中可以同时访问前端和后端代码，进行全栈分析

> 跨工作区引用文件使用 `@{工作区名}:path/to/file` 语法。

## 性能提示

| 建议 | 说明 |
|------|------|
| 控制并发数 | 日常使用建议 2~4 个会话，过多可能影响响应速度 |
| 及时清理 | 不再使用的会话及时关闭，释放内存 |
| 空闲自动释放 | Polaris 会自动释放长时间空闲会话的内存，切换时从快照恢复 |
| 内存快照上限 | 最多保留 20 个内存快照，超出会自动回收最旧的 |

## 快速切换面板

鼠标悬停在面板顶部的会话标识区域，会弹出快速切换面板：

- 显示所有会话列表（包含会话名称、消息数）
- 一键切换到任意会话
- 快速创建新会话
- 导出指定会话内容
