# 实战：代码审查工作流

本文演示如何利用 Polaris 的 AI 对话和 Git 集成进行高效的代码审查。

## 前提条件

- 已打开包含 Git 仓库的工作区
- 当前分支有待审查的代码变更

## Step 1：查看当前变更概览

1. 打开 **Git 面板** → **变更** 标签页
2. 浏览所有变更文件列表，了解改动范围
3. 点击关键文件查看差异对比

## Step 2：让 AI 分析代码变更

在 AI 对话中使用 Git 上下文：

```
请审查我当前的代码变更，重点关注：
1. 是否有潜在的 Bug 或逻辑错误
2. 代码风格是否一致
3. 是否有性能问题
4. 是否有安全隐患
```

AI 会通过 Git 上下文获取 diff 信息，逐文件分析并给出审查意见。

## Step 3：深入审查特定文件

如果 AI 发现了需要关注的问题，可以进一步追问：

```
请详细分析 src/utils/auth.ts 的变更，
这个改动是否会影响现有的登录流程？
有没有边界情况需要处理？
```

## Step 4：使用文件浏览器配合审查

1. 点击 ActivityBar 的 **Files** 图标
2. 找到 AI 提到的文件，双击打开
3. 在代码编辑器中查看完整上下文（不只是 diff 部分）
4. 结合 AI 的分析意见，确认是否需要修改

## Step 5：使用快捷片段提高效率

配置一个代码审查专用的快捷片段：

- **名称**：`review`
- **模板内容**：

```
请审查以下 {{language}} 代码变更，重点关注 {{focus_points}} 方面：

{{code}}

请给出：
1. 发现的问题（如果有）
2. 改进建议
3. 评分（1-5）
```

在 **设置 → 快捷片段** 中创建后，对话中输入 `/review` 即可快速调用。

## Step 6：根据审查意见修改

如果审查发现问题：

1. 让 AI 直接修复（配合权限审批）
2. 或手动在代码编辑器中修改
3. 修改后回到 Git 面板查看新的变更
4. 重复审查流程直到满意

## Step 7：提交审查通过的代码

1. 在 Git 面板暂存并提交变更
2. 使用 **AI 生成提交信息** 生成规范的 commit message
3. 推送到远程分支

## 高级技巧

### 审查他人的提交

在 Git 面板的 **历史** 标签页中查看他人的提交：

```
请审查最近一次提交的内容，评估代码质量和潜在风险
```

### 批量审查多个文件

```
请审查 src/api/ 目录下所有新增和修改的文件，
评估 API 设计是否合理，错误处理是否完善
```

### 关注安全审查

```
请对当前所有变更进行安全审查，特别关注：
- SQL 注入风险
- XSS 攻击风险
- 敏感信息泄露
- 权限校验是否完整
```
